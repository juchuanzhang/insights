核心问题

CPU提供架构级隔离（虚拟化、特权级、页表），但忽略了微架构侧信道：

  • 共享缓存、CPU缓冲区 → Meltdown、Foreshadow、MDS等攻击
  • 架构隔离 ≠ 微架构隔离——逻辑隔离无法阻止硬件级信息泄漏

微架构防御部署碎片化：

  • 硬件/固件层：部分由CPU补丁缓解
  • OS/hypervisor层：部分由软件缓解（缓冲区刷新、KPTI）
  • 未修补：安全影响低、缓解成本高 → 被忽略

补丁屡次失败

已知失败案例

  • Enhanced IBRS → BHI绕过
  • Automatic IBRS → Inception绕过
  • FineIBT → InSpectre Gadget绕过
  • retpoline → RETBLEED绕过
  • LFENCE/JMP → 仍可被竞争
  • VERW缓冲区覆写 → CacheOut绕过

验证手段不足

  • 依赖PoC攻击测试 → 覆盖面有限
  • 依赖纯理论推理 → 易出错、劳动密集
  • 厂商安全声明屡被推翻
  • 没有自动化、系统性的跨域验证工具

核心方法论：Actor抽象

引入 Actor 作为安全域的统一抽象：

Actor = 代码 + 私有数据 + 特权级 + CPU模式 + 内存权限
  • Actor = 进程 / VM / 内核 / 任何隔离执行上下文
  • Actor之间通过预定义指令序列（syscall、VM enter）进行域转换
  • 每个test case = 1个main Actor(内核host) + 1~N个其他模式Actor

测试目标：域转换时，受害者Actor的数据不应泄漏到攻击者Actor

模板：汇编文件，定义Actor结构和域转换边界

执行环境：3500+行定制内核模块

为什么不用Linux现有基础设施？

  • Linux VM管理面向完整VM → 开销过大
  • Linux有大量安全/稳定性防护 → 阻止自由配置
  • 因此从零实现定制内核模块

关键设计

  • 状态保存恢复 保存/恢复CR0、EFER、VMX寄存器
  • 故障隔离 捕获所有异常→自定义处理器
  • CPU配置 通过MSR控制预取器等特性
  • VM创建 毫秒级创建VMCS/VMCB+EPT/NPT
  • 内存别名 相同虚拟地址布局模拟Foreshadow

泄漏建模：合同式非干扰

经典非干扰（太强）：受害者数据σ^V不应影响攻击者测量

σ^1_A = σ^2_A ⇒ Measure(p, σ^1, μ) = Measure(p, σ^2, μ)

合同式非干扰（实用化）：允许基本泄漏，仅禁止超出预期的泄漏

σ^1_A = σ^2_A ∧ Contract(p, σ^V_1) = Contract(p, σ^V_2) ⇒ Measure(p, σ^1, μ) = Measure(p, σ^2, μ)

  • 合同轨迹 = 模型预测的预期泄漏（受害者load/store地址 + 控制流变化）
  • 实际OS允许基本泄漏（如sysret后不刷缓存），但不允许MDS类深层泄漏
  • 隔离仅在泄漏超出预期时才被违反

非确定性处理：χ²检验 + 自适应样本量

问题：域转换触发不透明微码，引入噪声 → 假阳性频发

解决方案：将轨迹相等比较提升为分布相等比较

  • 硬件轨迹视为类别数据 → Pearson χ²检验
  • 两条轨迹"相同" iff 它们来自同一分布
  • 自适应样本量：N=15 → 40 → 160 → 320
  • 仅在最大样本量仍持续违反时才报告

新发现①：跨VM泄漏

CVE-2024-36357 潜在高影响 AMD3 (Zen4) V2V / MEM W-bit

机制：攻击者VM加载地址时，若缓存中含受害VM相同虚拟地址（不同host物理地址）的值 → 受害者缓存值影响攻击者后续操作时序 → 逐位推断受害VM任意内存

a = *p1;           // p1别名指向受害VM地址
array1[a] = 0;
*p2 = 0;           // 页故障（写只读页）
...                 // 算术操作序列
*p3 = 0;           // 页故障
c = *p5;           // 任意有效地址
d = array2[c];     // 缓存暴露点

根因尚未公开——AMD将在安全公告中披露。gadget可能只是众多触发方式之一。

新发现②：内核→用户泄漏

CVE-2024-36350 中高影响 AMD3 (Zen4) K2U / MEM U-bit

机制：用户进程可观察到最近 N≈32 次内核态store操作的任意一位

  • 即使store在内核态执行，用户态gadget的时序仍依赖其数据
  • 攻击者可通过额外store选择目标store，通过移位选择泄漏位
  • gadget类似跨VM泄漏，但依赖用户态访问内核内存触发的异常

影响：攻击者可逐位提取特权数据——观察切换到用户态前的内核store值

新发现③④：AMD RSRR声明被反驳

③ RDTSCP-AUX特权读取

CVE-2024-36349 低影响
  • 影响: Int1~3 + AMD1
  • 用户态可推测性读取RDTSCP AUX寄存器，即使CR4.TSD已设置
  • AUX通常仅含CPU ID → 安全影响低
  • 反驳AMD白皮书声称不受RSRR影响
  • AMD不计划修复

④ SMSW推测执行

CVE-2024-36348 低影响但功能违反
  • 影响: AMD2 + AMD3
  • 用户态推测性执行SMSW，即使UMIP已启用 → 返回CR0低16位
  • 实质上绕过了UMIP功能
  • 同样反驳AMD RSRR声明
  • AMD不计划修复

关键发现：AMD白皮书声称其CPU不受RSRR影响 → 本文在AMD CPU上发现两类RSRR → 声明被实证反驳

FAQ

Q1：为什么不测TDX/SEV-SNP？

  • Actor模式不支持SEAM模式（TDX新增特权级）
  • 信任模型相反：论文hypervisor可信，CVM中hypervisor不可信
  • 执行器缺SEAM/RMP等CVM基础设施
  • 2024年TDX/SEV-SNP硬件可用性有限
  • SEC类指令已在REG池中但仅作RSRR测试，非完整CVM域转换

Q2：缓解措施作为"特定指令序列"怎么理解？

  • VERW/L1D_FLUSH/DSS补丁都是每次域转换前必须实际执行的指令，不是配置位开关
  • 必须在域转换紧前方执行才有意义（否则buffer可能被重新填充）
  • 模板机制允许灵活插入缓解序列在不同位置

Q3：测了哪些防护措施？

7种补丁变体（VERW ✓、L1D_FLUSH ✓、WBINVD ✗×2、DSS当前 ✓、DSS旧 ✗、KPTI×3），3种无效 → 证明没有自动化验证无法确认补丁有效

关键结论

  1. 微架构隔离验证亟需自动化工具 — 被动式补丁验证屡次失败
  2. Actor + MRT 提供了系统性跨域微架构泄漏检测的可行路径
  3. 4个新CVE 证明工具有效（2高影响 + 2反驳厂商声明）
  4. AMD RSRR声明被实证反驳 — 在AMD CPU上发现两类RSRR漏洞
  5. 补丁验证价值显著 — 3种补丁无效（含差点合入主线的DSS补丁）
  6. 从被动补丁到主动验证的范式转变 是处理器安全设计的未来方向
