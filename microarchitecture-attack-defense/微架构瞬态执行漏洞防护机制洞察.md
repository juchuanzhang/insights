## 1. 引言

### 1.1 背景

2018年1月，Google Project Zero 团队公开披露了 Spectre 和 Meltdown 两类利用现代处理器推测执行（Speculative Execution）机制的侧信道攻击方法[[1]](https://spectreattack.com/spectre.pdf)。这些攻击利用处理器在推测执行过程中留下的微架构状态痕迹（如缓存、分支预测器状态、行填充缓冲器等），使得非特权攻击者能够窃取受保护的内存数据，从根本上动摇了现代处理器性能优化与安全隔离之间的平衡[[2]](https://meltdownattack.com/meltdown.pdf)。

此后五年间，学术界和工业界陆续发现了数十种瞬态执行（Transient Execution）攻击变体，覆盖了几乎所有主流高性能处理器架构。本文将系统性地梳理 Intel、AMD、ARM 三大商用处理器平台的硬件级防御机制、Linux 内核的软件防御体系，以及编译器、虚拟化平台、Web 浏览器等关键软件的防护方案，所有内容均标注参考文献来源。

### 1.2 核心定义

**瞬态执行（Transient Execution）**：指处理器在推测路径上执行的指令序列，这些指令的架构效果最终会被丢弃（不会被提交到架构状态），但其微架构副作用（如缓存状态变化、TLB 填充、预测器训练等）会持续存在，从而可被攻击者观测和利用[[3]](https://arxiv.org/abs/1811.05441)。

**微架构侧信道（Microarchitectural Side Channel）**：利用处理器内部微架构组件的共享特性或状态残留，通过测量时间差异等方式推断隐秘信息的攻击通道[[4]](https://doi.org/10.1007/s13389-017-0159-4)。

### 2.2 按攻击目标分类

- **跨特权级攻击**：用户态 → 内核态 / 用户态 → Hypervisor（如 Meltdown、L1TF、MDS）
- **跨进程攻击**：进程 A → 进程 B（如 Spectre v2、BHI）
- **跨超线程攻击**：SMT 同核线程之间的攻击（如 MDS、Spectre v2 + SMT）
- **跨 VM 攻击**：Guest → Host / Guest → Guest（如 L1TF、Spectre v2）
- **SGX 飞地攻击**：针对 Intel SGX 可信执行环境的攻击（如 LVI、SGAxe）

## 3. Intel 商用处理器防御机制

### 3.1 微码更新的新能力

Intel 通过微码更新向现有处理器添加了多项硬件级防御功能。自 2018 年起，Intel 累计发布了超过 30 个安全相关的微码更新[[23]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/best-practices/microcode-update-guidance.html)。

#### 3.1.1 IA32_SPEC_CTRL MSR (MSR 0x48)

处理器通过 `IA32_SPEC_CTRL` 模型特定寄存器控制推测执行行为，该寄存器包含以下关键位[[24]](https://www.intel.com/sdm)：

| 位 | 名称 | 功能 | 引入版本 |
|----|------|------|----------|
| Bit 0 | **IBRS** (Indirect Branch Restricted Speculation) | 限制间接分支推测，防止跨特权级 BTB 污染 | 2018 年微码[[23]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/best-practices/microcode-update-guidance.html) |
| Bit 1 | **STIBP** (Single Thread Indirect Branch Predictors) | 在 SMT 环境下隔离间接分支预测器 | 2018 年微码[[23]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/best-practices/microcode-update-guidance.html) |
| Bit 2 | **SSBD** (Speculative Store Bypass Disable) | 禁用推测性存储绕过优化 | 2018 年微码[[23]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/best-practices/microcode-update-guidance.html) |

**IBRS 的三种工作模式**[[25]](https://www.intel.com/content/dam/www/public/us/en/documents/white-papers/analysis-of-speculative-execution-side-channels-white-paper.pdf)：

1. **Legacy IBRS**（传统模式）：
   - 进入内核时设置 IBRS=1，退出时清空 IBRS=0
   - 需要与 retpoline 配合使用
   - 每次系统调用/中断需要在 MSR 写入上产生性能开销
   - 适用于 Skylake 及更早架构

2. **Enhanced IBRS (eIBRS)**（增强模式）：
   - 一次设置 IBRS=1 后不再清空
   - 硬件自动在用户态返回时隔离 BTB
   - 自动包含 STIBP 功能（跨线程保护）
   - 性能开销大幅低于 Legacy IBRS
   - 适用于 Cascade Lake 及更新架构（需 CPUID.(EAX=7H,ECX=0):EDX[[29]](https://docs.kernel.org/admin-guide/hw-vuln/spectre.html)=1）[[26]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/enhanced-indirect-branch-restricted-speculation.html)
   - **限制**：eIBRS 不能防御 BHI 攻击，需要 BHI_DIS_S 或软件序列[[14]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/branch-history-injection.html)

3. **Automatic IBRS (Auto-IBRS)**：
   - 与 eIBRS 类似，由硬件自动管理 IBRS 位
   - 目前主要见于 AMD 处理器

#### 3.1.2 IA32_PRED_CMD MSR (MSR 0x49)

用于发出预测器刷新命令[[24]](https://www.intel.com/sdm)：

| 位 | 名称 | 功能 |
|----|------|------|
| Bit 0 | **IBPB** (Indirect Branch Prediction Barrier) | 触发间接分支预测器刷新 |

IBPB 在以下场景使用[[27]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/indirect-branch-prediction-barrier.html)：
- 上下文切换到敏感进程时
- 虚拟机切换时（vmexit 处理）
- 进程通过 `prctl()` 主动请求隔离时

#### 3.1.3 IA32_ARCH_CAPABILITIES MSR (MSR 0x10A)

此只读 MSR 报告处理器的安全能力，关键位包括[[24]](https://www.intel.com/sdm)[[28]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/ia32-arch-capabilities-msr.html)：

| 位 | 名称 | 含义 |
|----|------|------|
| Bit 0 | **RDCL_NO** | 对 Meltdown 免疫（不出现 Rogue Data Cache Load） |
| Bit 1 | **IBRS_ALL** | 支持 Enhanced IBRS 模式 |
| Bit 2 | **RSBA** (Return Stack Buffer Alternate) | Retpoline 可能需要额外 RSB 填充 |
| Bit 3 | **SKIP_L1DFL_VMENTRY** | 可跳过 L1D 刷新（对 L1TF 免疫） |
| Bit 4 | **SSB_NO** | 对 Speculative Store Bypass 免疫 |
| Bit 5 | **MDS_NO** | 对 MDS 攻击免疫 |
| Bit 6 | **IF_PSCHANGE_MC_NO** | 对微码导致的预测器变化免疫 |
| Bit 7 | **TSX_CTRL** | 支持 TSX 控制 MSR |
| Bit 8 | **TAA_NO** | 对 TSX Asynchronous Abort 免疫 |
| Bit 10 | **PSCHANGE_MC_NO** | 对预测器变化触发性能监视中断免疫 |
| Bit 13 | **SBDR_SSDP_NO** | 对 Shared Buffer Data Read 免疫 |
| Bit 14 | **FBSDP_NO** | 对 Fill Buffer Stale Data Propagation 免疫 |
| Bit 15 | **PSDP_NO** | 对 Primary Stale Data Propagation 免疫 |
| Bit 17 | **FB_CLEAR** | 支持 Fill Buffer 清除指令（VERW） |
| Bit 18 | **FB_CLEAR_CTRL** | 支持 Fill Buffer 清除控制 |
| Bit 19 | **RRSBA** | 需要增强的 RSB 替代预测和 BHI 缓解 |
| Bit 20 | **BHI_NO** | 对 Branch History Injection 免疫 |
| Bit 21 | **XAPIC_DISABLE_STATUS** | 支持 xAPIC 禁用状态 |
| Bit 22 | **PREFETCHITI_CTRL** | 支持 PREFETCHITI 控制 |
| Bit 23 | **GDS_NO** | 对 Gather Data Sampling 免疫 |
| Bit 24 | **RFDS_NO** | 对 Register File Data Sampling 免疫 |
| Bit 25 | **MCDT_NO** | 对 MXCSR Configuration Dependent Timing 免疫 |

#### 3.1.4 硬件特性：BHI_DIS_S

BHI_DIS_S (Branch History Injection Disable via Software) 是于 2022 年由 Intel 推出的硬件缓解特性[[14]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/branch-history-injection.html)。当启用时，处理器将禁用分支历史缓冲区（BHB）对间址预测的影响。

- 通过 `IA32_SPEC_CTRL` 启用的相关控制位
- 仅在支持 eIBRS 且 IA32_ARCH_CAPABILITIES.BHI_NO=0 的处理器上需要
- 性能影响：测试显示 0-15% 的性能下降，取决于工作负载的分支密集程度[[29]](https://docs.kernel.org/admin-guide/hw-vuln/spectre.html)
- 适用型号：Alder Lake、Raptor Lake、Sapphire Rapids 及更新架构[[14]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/branch-history-injection.html)

#### 3.1.5 硬件特性：IPRED (Indirect Predictor)

于 2024 年推出的增强间接预测器，具有以下安全特性[[30]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/indirect-predictor.html)：
- 间接分支预测器与直接分支预测器硬件隔离
- 使用模式标签（Modality Tags）防止跨特权级预测器污染
- 缓解 ITS（Indirect Target Selection）和 IBPD（Indirect Branch Predictor Delayed Updates）攻击

#### 3.1.6 硬件特性：Data-Dependent Prefetcher (DDP)

较新的 Intel 处理器（Granite Rapids 起）支持数据依赖预取器，其设计特性可限制依赖预取被用作侧信道利用[[31]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/data-dependent-prefetcher.html)：
- 预取决策仅基于数据流依赖而非控制流预测
- 限制攻击者对预取路径的控制能力

#### 3.1.7 指令：VERW (VERify Word)

Intel 使用 `VERW` 指令的微码实现来清除内部 CPU 缓冲区[[32]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/cpuid-enumeration-and-architectural-msrs.html)：
- 清除填充缓冲区 (Fill Buffers)
- 清除加载端口缓冲区 (Load Port Buffers)
- 清除存储缓冲区 (Store Buffers)
- 需要 IA32_ARCH_CAPABILITIES.FB_CLEAR=1
- 用于 MDS、TAA、RFDS 和 SRBDS 的缓解

#### 3.1.8 硬件特性：Fast Store Forwarding Predictor (FSFP)

Intel 在新平台上引入了 Fast Store Forwarding Predictor，其设计保障[[33]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/fast-store-forwarding-predictor.html)：
- 写转发预测仅基于真实完成的存储-加载对
- 不会基于推测执行的写操作来训练预测器
- 防止了利用存储转发预测器的侧信道攻击

## 4. AMD 商用处理器防御机制

### 4.1 AMD 对推测执行的控制架构

AMD 处理器采用了与 Intel 不同但功能对等的推测控制机制。AMD 在 IEEE/ACM 国际架构会议上发布了其推测控制白皮书[[34]](https://www.amd.com/content/dam/amd/en/documents/processor-tech-docs/white-papers/111006-architecture-guidelines-update-amd64-technology-indirect-branch-control-extension.pdf)。

#### 4.1.1 AMD 的 MSR 实现

AMD 实现了与 Intel 兼容的 MSR 接口，但部分行为存在差异[[34]](https://www.amd.com/content/dam/amd/en/documents/processor-tech-docs/white-papers/111006-architecture-guidelines-update-amd64-technology-indirect-branch-control-extension.pdf)[[35]](https://www.amd.com/en/support/tech-docs/amd64-architecture-programmers-manual-volume-2-system-programming)：

| MSR | AMD 位定义 | 与 Intel 的关键差异 |
|-----|-----------|---------------------|
| **MSR 0x48** (SPEC_CTRL) | Bit 0: IBRS, Bit 1: STIBP, Bit 2: SSBD, Bit 3: PSFD | AMD PSFD (Predictive Store Forwarding Disable)[[36]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-1037.html) |
| **MSR 0x49** (PRED_CMD) | Bit 0: IBPB | 行为相似，但 AMD 的 IBPB 本身不包括 RSB 刷新 |
| **MSR 0x10A** (ARCH_CAPABILITIES) | 不同位定义 | AMD 使用不同的能力报告结构 |

#### 4.1.2 AMD IBPB 的特殊行为

AMD 处理器的 IBPB 执行具有以下特点[[37]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-1031.html)：
- **RSB 刷新**：AMD 的 IBPB 指令默认**不刷新**返回栈缓冲区 (RSB)，与 Intel 的行为关键不同。需要在 IBPB 后显式执行 RSB 填充序列
- **Inception 缓解**：为解决 CVE-2023-20569，AMD 微码更新使 IBPB 执行后也清除分支预测器中的返回地址预测状态[[18]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-7005.html)
- **性能影响**：AMD IBPB 的执行开销约 200-400 cycles（Zen 3 平台测试）[[37]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-1031.html)

#### 4.1.3 AMD 的技术白皮书：推测管理的软件技术

AMD 发布了详细的《Software Techniques for Managing Speculation on AMD Processors》白皮书[[34]](https://www.amd.com/content/dam/amd/en/documents/processor-tech-docs/white-papers/111006-architecture-guidelines-update-amd64-technology-indirect-branch-control-extension.pdf)，建议的防御策略包括：

1. **LFENCE 作为推测屏障**：
   - AMD 始终将 LFENCE 指令视为序列化指令（自 2018 年起在文档中明确）
   - 编译器可在条件分支后插入 LFENCE 防止 Spectre v1
   - 与 Intel LFENCE 不同，AMD LFENCE 在推测路径上也阻止后续指令发射[[34]](https://www.amd.com/content/dam/amd/en/documents/processor-tech-docs/white-papers/111006-architecture-guidelines-update-amd64-technology-indirect-branch-control-extension.pdf)

2. **间接分支控制扩展 (IBX)**：
   - AMD 通过微码在 Zen 2 及更新的架构上支持完整的 IBRS/STIBP/IBPB
   - Zen 1 需要 AGESA 固件更新（如 ComboAM4PI 1.0.0.8）以获得完整支持[[38]](https://www.amd.com/en/resources/support-articles/faqs/AGESA-Update.html)

3. **Predictive Store Forwarding Disable (PSFD)**：
   - AMD Zen 3 及以上特性，用于防止利用预测性存储转发的 Spectre v4 变种
   - 通过 MSR 0x48 Bit 3 控制[[36]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-1037.html)

### 4.3 Zenbleed 的缓解细节 (CVE-2023-20593)

Zenbleed 是影响 AMD Zen 2 微架构（Ryzen 3000/4000/5000 系列及 EPYC 7002）的寄存器泄漏漏洞[[17]](https://lock.cmpxchg8b.com/zenbleed.html)。

**攻击原理**：在 Zen 2 处理器中，当 XMM 寄存器重命名发生特定竞争条件时，错误预测的 VZEROUPPER 指令可能导致当前寄存器文件内容通过 `vzeroupper` 的优化实现泄漏到相邻逻辑线程[[17]](https://lock.cmpxchg8b.com/zenbleed.html)。

**硬件缓解**：
```asm
; 通过设置 DE_CFG MSR (MSR 0xC0011029) Bit 9 来缓解
; 这强制 VZEROUPPER 以更保守的管道刷新方式执行
; 操作需在 BIOS/UEFI 层面通过 AGESA 更新实现
; ComboAM4v2 PI 1.2.0.A 及更新的 AGESA 版本包含修复
```

**性能影响**：AMD 评估在大多数工作负载下可忽略不计[[39]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-7008.html)。

### 4.4 Inception 的缓解细节 (CVE-2023-20569)

Inception 影响 Zen 3 和 Zen 4 微架构，利用返回指令的预测机制实现推测性控制流劫持[[18]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-7005.html)。

**攻击原理**：攻击者训练分支预测器，使 `RET` 指令在推测路径上跳转到任意地址，类似于 Spectre v2 但针对返回预测机制。在 AMD 处理器中，返回预测器使用独立的返回地址预测栈，但这仍可被训练[[18]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-7005.html)。

**微码缓解**：
- **增强 IBPB**：IBPB 执行后，额外清除与返回地址预测相关的内部缓冲区
- **RSB 深度扩展**：增加硬件 RSB 深度以增加训练难度
- **IBRS 增强**：在 IBRS=1 时，返回预测也不跨特权级
- 通过 AGESA 更新部署（如 ComboAM4v2 PI 1.2.0.B）[[40]](https://www.amd.com/en/resources/support-articles/release-notes/AGESA-PI-1.2.0.B.html)

**软件缓解**（如无法更新微码）：
- 在关键代码路径的 RET 指令前插入 IBPB
- RSB 填充序列在上下文切换时执行 32 次而非 16 次

### 4.5 AMD 固件层面的安全检查点

AMD 通过 AGESA™ (AMD Generic Encapsulated Software Architecture) 固件包提供固件级安全修复[[38]](https://www.amd.com/en/resources/support-articles/faqs/AGESA-Update.html)：

- **ComboAM4PI 1.0.0.8**：提供完整的 Spectre v2 缓解支持
- **ComboAM4v2 PI 1.2.0.4**：添加推测执行控制的完整 MSR 支持
- **ComboAM4v2 PI 1.2.0.8**：增强安全相关配置
- **ComboAM4v2 PI 1.2.0.A**：Zenbleed (CVE-2023-20593) 修复[[39]](https://www.amd.com/en/resources/product-security/bulletin/amd-sb-7008.html)
- **ComboAM4v2 PI 1.2.0.B**：Inception (CVE-2023-20569) 修复[[40]](https://www.amd.com/en/resources/support-articles/release-notes/AGESA-PI-1.2.0.B.html)

## 5. ARM 商用处理器防御机制

### 5.1 ARM 架构的推测控制原语

ARM 在其架构规范中定义了一组用于推测控制的系统寄存器[[41]](https://developer.arm.com/documentation/102816/latest/)。

#### 5.1.1 CSV2 (Cache Speculation Variant 2) 机制

ARMv8.0 及以上架构支持的主要推测控制特性[[41]](https://developer.arm.com/documentation/102816/latest/)[[42]](https://developer.arm.com/documentation/ddi0601/)：

| 特性 | 系统寄存器 | 功能描述 | 引入架构 |
|------|-----------|----------|----------|
| **CSV2** | ID_AA64PFR0_EL1.CSV2 | 硬件支持 Spectre v2 的 SCXTNUM 寄存器 | ARMv8.0+ (可选) |
| **CSV2_1p1** | ID_AA64PFR0_EL1.CSV2>=2 | 硬件隔离 1 级和 0 级间接预测器 | ARMv8.2+ |
| **CSV2_1p2** | ID_AA64PFR0_EL1.CSV2>=3 | 硬件隔离所有特权级间接预测器 | ARMv8.2+ |
| **CSV2_2** | ID_AA64PFR0_EL1.CSV2>=4 | 增强版预测器隔离 | ARMv8.4+ |
| **CSV2_3** | ID_AA64PFR0_EL1.CSV2>=5 | 最全面的硬件隔离 | ARMv8.5+ |
| **CSV3** | ID_AA64PFR0_EL1.CSV3 | 对 Meltdown 硬件免疫 | ARMv8.2+ (可选) |

**CSV2 工作机制**[[42]](https://developer.arm.com/documentation/ddi0601/)：
- CSV2>=1：引入 `SCXTNUM_EL0` 和 `SCXTNUM_EL1` 寄存器，允许操作系统在每个特权级标记预测上下文
- CSV2>=2：硬件自动在 EL0↔EL1 切换时清除间接预测器
- CSV2>=3：硬件自动在所有特权级切换时清除间接预测器
- **无需软件配合**即可防御 Spectre v2 跨特权级攻击

#### 5.1.2 Speculation Barrier (SB) 指令

ARMv8.0 引入的推测屏障指令[[43]](https://developer.arm.com/documentation/ddi0487/)：

```asm
SB    ; Speculation Barrier - 阻止后续指令在 SB 之前的条件解析前执行
```

- 用于防止 Spectre v1 类型的攻击
- 在 `dsb nsh` 和 `isb` 之后也隐式包含推测屏障效果
- 使用场景：数组边界检查后、指针验证后

ARMv8.5 将 `SB` 从可选特性提升为强制特性[[44]](https://developer.arm.com/Architectures/A-Profile%20Architecture)。

#### 5.1.3 SSBD (Speculative Store Bypass Disable)

ARMv8.0 引入的特性[[41]](https://developer.arm.com/documentation/102816/latest/)：

| 寄存器 | 位 | 功能 |
|--------|----|------|
| SSBD (PSTATE) | 直接设置 | 禁用推测性存储绕过 |
| SSBS (PSTATE) | Bit 12 | 推测性存储绕过安全状态 |

**三种工作方式**：
1. **传统 SSBD**：通过系统寄存器写控制
2. **SSBS (Speculative Store Bypass Safe)**：PSTATE 位，每个异常级别独立控制，开销更低
3. **硬件自动修复**：ID_AA64PFR1_EL1.SSBS=2 表示硬件直接修复，无需软件干预[[45]](https://developer.arm.com/documentation/ddi0487/)

#### 5.1.4 FEAT_CSV2_1p1 和 FEAT_CSV2_1p2

关键硬件进展[[46]](https://developer.arm.com/documentation/102816/latest)：

- **FEAT_CSV2_1p1** (ARMv8.2+)：
  - 硬件自动在 EL0↔EL1 切换时清除分支预测器状态
  - 交叉特权级的 Spectre v2 完全无需内核干预
  - 适用于 Cortex-A76/A77/A78 及更新设计

- **FEAT_CSV2_1p2** (ARMv8.2+)：
  - 硬件在所有特权级切换（含 EL1↔EL2）时清除
  - 适用于 Cortex-A78C/X1 及更新设计

#### 5.1.5 FEAT_BTI (Branch Target Identification)

ARMv8.5-A 引入的硬件加强特性[[47]](https://developer.arm.com/documentation/ddi0487/)：

- 在间接分支目标地址处插入 `BTI` 指令（编码为 `hint #34` 等）
- 处理器在间接分支后检查目标地址是否包含合法的 `BTI` 标记
- 防止面向跳转编程（JOP）攻击
- 特别是防御 Spectre v2 类型的控制流劫持

```asm
; BTI 使用示例
function_with_bti:
    BTI C          ; 标记为合法间接调用目标
    ; ... 函数体 ...
    RET
```

#### 5.1.6 FEAT_PAC (Pointer Authentication)

ARMv8.3-A 引入的指针认证[[48]](https://developer.arm.com/documentation/ddi0487/)：

- 使用 PAC 指令对指针进行密码学签名
- 防止面向返回编程（ROP）攻击
- 间接有助于防止利用 Spectre v2 的类型混淆

```asm
PACIASP           ; 使用 SP 作为上下文认证 LR 的值
; ... 函数体 ...
AUTIASP           ; 验证 LR 的签名
RET               ; 返回
```

#### 5.1.7 FEAT_ETS (Enhanced Translation Synchronization)

ARMv8.1+ 特性，虽然不是直接用于瞬态执行防御，但其 TLB 失效机制的加强有助于防止利用 TLB 侧信道的攻击[[49]](https://developer.arm.com/documentation/ddi0487/)。

## 6. Linux 内核防御机制

### 6.1 内核页表隔离 (KPTI / KAISER)

**漏洞**：Meltdown (CVE-2017-5754) [[5]](https://nvd.nist.gov/vuln/detail/CVE-2017-5754)

**机制**[[51]](https://gruss.cc/files/kaiser.pdf)：
- 将内核页表与用户态页表分离
- 用户态运行时，仅映射极少的内核入口代码（trampoline）
- 系统调用/中断时切换至完整内核页表
- 使用 `CR3` 写入在用户态和内核态页表之间切换

**x86 实现**[[52]](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/arch/x86/entry/entry_64.S)：
```c
// KPTI 的关键路径
// 进入内核时切换到内核页表
movq    PER_CPU_VAR(cpu_tss_rw + TSS_sp0), %rsp
// 切换到内核 CR3
movq    %cr3, %r14
movq    PER_CPU_VAR(cpu_tlbstate + TLBSTATE_user_pcid_flush_mask), %r15
andq    %r14, %r15
orq     PER_CPU_VAR(cpu_tlbstate + TLBSTATE_kernel_cr3), %r15
movq    %r15, %cr3
```

**内核命令行控制**[[53]](https://docs.kernel.org/admin-guide/kernel-parameters.html)：
- `pti=on`：启用 KPTI（默认对受影响处理器开启）
- `pti=off`：禁用 KPTI
- `nopti`：同 `pti=off`
- `pti=auto`：自动检测（默认行为）

**性能影响**[[54]](https://www.redhat.com/en/blog/performance-implications-spectre-and-meltdown-mitigations)：
- Intel 旧款 (Skylake and earlier)：5-30% 的性能下降（I/O 密集型负载影响最大）
- Intel 新款 (Cascade Lake+)：0-3%（硬件修复 + PCID 优化）
- AMD 不受影响：自动检测并禁用 KPTI

**PCID 优化** (Process-Context Identifier)[[55]](https://www.intel.com/content/www/us/en/developer/articles/technical/intel-sdm.html)：
- Intel Westmere 起支持 PCID
- KPTI 利用 PCID 避免在用户态/内核态切换时完全刷新 TLB
- 大幅降低 KPTI 的性能开销（从 30% 降至 5-10%）
- 通过 `nopcid` 命令行参数禁用

### 6.2 Retpoline (返回跳板)

**漏洞**：Spectre v2 (CVE-2017-5715) [[1]](https://spectreattack.com/spectre.pdf)

**原理**[[56]](https://support.google.com/faqs/answer/7625886)：
- 使用 `RET` 指令实现间接跳转，替代 `JMP *%reg` / `CALL *%reg`
- 利用 `RET` 使用返回栈缓冲区 (RSB) 预测而非分支目标缓冲区 (BTB)
- 在推测执行时陷入无限循环，防止跳转到恶意 gadget

**内核实现**[[57]](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/arch/x86/lib/retpoline.S)：

```asm
; x86-64 retpoline 的核心 thunk
__x86_indirect_thunk_\reg:
    call    111f
    111:
    pause                           ; lfence 在某些变体中
    lfence                          ; AMD 首选序列化
    jmp     111b                    ; 捕获推测执行
    mov     %\reg, (%rsp)           ; 非推测路径
    RET
```

**编译选项**[[58]](https://gcc.gnu.org/onlinedocs/gcc/x86-Options.html)：
```bash
# GCC
-mindirect-branch=thunk-extern
-mindirect-branch-register
-mfunction-return=thunk-extern

# Clang/LLVM
-mretpoline-external-thunk
```

**配置**[[59]](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/arch/x86/Kconfig)：
```
CONFIG_MITIGATION_RETPOLINE=y
```

**内核命令行控制**[[53]](https://docs.kernel.org/admin-guide/kernel-parameters.html)：
- `spectre_v2=retpoline`：强制使用 retpoline
- `spectre_v2=auto`：自动选择最佳策略
- `spectre_v2=off`：禁用所有 Spectre v2 缓解

**性能影响**[[60]](https://doi.org/10.1109/MICRO.2018.00032)：
- 基础 retpoline：3-10% 性能下降（分支密集型代码影响更大）
- 现代 CPU + RSB 填充优化：2-5%
- 与 eIBRS 相比：retpoline 影响更大因为每次调用都有开销

### 6.3 返回栈缓冲区 (RSB) 填充

**漏洞**：Spectre v2 变种（通过 RSB 下溢）[[61]](https://www.usenix.org/system/files/conference/woot18/woot18-paper-koruyeh.pdf), Retbleed (CVE-2022-29900/29901) [[15]](https://comsec.ethz.ch/research/microarch/retbleed/)

**机制**[[15]](https://comsec.ethz.ch/research/microarch/retbleed/)：
- 在上下文切换、VMExit、进入深层调用栈时主动填充 RSB
- 防止 RSB 下溢时回退到 BTB 预测
- 防止跨进程 RSB 中毒

**Retbleed 特定缓解**[[15]](https://comsec.ethz.ch/research/microarch/retbleed/)[[62]](https://docs.kernel.org/admin-guide/hw-vuln/retbleed.html)：
- Intel Skylake 一代：eIBRS 不足以防御，需要 RSB 填充（因为 RSBA=1）
- AMD Zen 1/2/3：需要使用 IBRS (即使 Linux 之前默认使用 retpoline)
- Zen 1/2 还需配合 STIBP + IBPB
- Zen 4：硬件修复，RSBA=0

**内核实现**[[63]](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/arch/x86/include/asm/nospec-branch.h)：
```c
// 上下文切换时填充 RSB
#define RSB_CLEAR_LOOPS 32  // Zenbleed/Inception 之后增加至 32
for (i = 0; i < RSB_CLEAR_LOOPS; i++) {
    alternative("call 1f; int3; 1:", 
                "call 2f; pause; lfence; jmp 2f; 2:", 
                X86_FEATURE_RSB_CTXSW);
}
```

### 6.4 Speculative Store Bypass Disable (SSBD)

**漏洞**：Spectre v4 (CVE-2018-3639) [[7]](https://nvd.nist.gov/vuln/detail/CVE-2018-3639)

**硬件接口**[[64]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/speculative-store-bypass.html)：
```
IA32_SPEC_CTRL Bit 2 (SSBD)   Intel
MSR 0x48 Bit 2 (SSBD)         AMD
PSTATE.SSBS                   ARM
```

**用户态控制**[[65]](https://docs.kernel.org/userspace-api/spec_ctrl.html)：
```c
// 用户程序可以通过 prctl() 控制自身 SSBD 行为
prctl(PR_SET_SPECULATION_CTRL,
      PR_SPEC_STORE_BYPASS,
      PR_SPEC_ENABLE,    // 禁用推测存储绕过 = 启用保护
      0, 0);

prctl(PR_SET_SPECULATION_CTRL,
      PR_SPEC_STORE_BYPASS,
      PR_SPEC_DISABLE,   // 允许推测存储绕过 = 禁用保护
      0, 0);
```

**内核命令行**[[53]](https://docs.kernel.org/admin-guide/kernel-parameters.html)：
```
spec_store_bypass_disable=on     # 全局启用
spec_store_bypass_disable=off    # 全局禁用
spec_store_bypass_disable=auto   # 仅 seccomp 进程启用
spec_store_bypass_disable=prctl  # 进程通过 prctl 自行控制
```

### 6.5 L1D Flush (L1 数据缓存刷新)

**漏洞**：L1TF/Foreshadow (CVE-2018-3615/3620/3646) [[8]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/l1-terminal-fault.html)

**机制**[[66]](https://docs.kernel.org/admin-guide/hw-vuln/l1tf.html)：
1. **PTE 反转 (PTE Inversion)**：
   - 当页表项 (PTE) 标记为不存在时，内核将 PTE 的地址指向一个可安全访问的页
   - 防止攻击者通过 L1TF 推测访问指向敏感物理地址的 PTE

2. **L1D 刷新**[[67]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/l1d-flush.html)：
   - 进入 VM 前刷新 L1 数据缓存
   - 使用 `L1D_FLUSH` MSR 写入（Intel）或 `wbinvd` 系列指令
   - x86 汇编序列：`mov $0x1F, %eax; xor %edx, %edx; wrmsr`

3. **核心调度 (Core Scheduling)**[[68]](https://docs.kernel.org/admin-guide/hw-vuln/core-scheduling.html)：
   - 启用 `CONFIG_SCHED_CORE` 配置
   - 确保互不信任的进程/VM 不在同一核心的 SMT 线程上运行
   - 内核命令行参数：`l1tf=full,force`

**内核命令行**[[53]](https://docs.kernel.org/admin-guide/kernel-parameters.html)：
```
l1tf=off          # 禁用所有 L1TF 缓解
l1tf=flush        # 仅在 VM entry 时刷新 L1D
l1tf=full         # 完整缓解 (flush + PTE 反转)
l1tf=full,force   # 完整缓解 + 强制禁用 SMT
```

### 6.6 CPU 缓冲区清除 (MDS / TAA / RFDS / SRBDS)

**漏洞**：MDS (CVE-2018-12126/12127/12130, CVE-2019-11091) [[9]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/microarchitectural-data-sampling.html), TAA (CVE-2019-11135) [[11]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/tsx-asynchronous-abort.html), RFDS (CVE-2023-28746) [[19]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/register-file-data-sampling.html), SRBDS (CVE-2020-0543) [[13]](https://www.intel.com/content/www/us/en/developer/topic-technology/software-security-guidance/advisory-guidance/special-register-buffer-data-sampling.html)

**统一缓解**[[32]](https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/cpuid-enumeration-and-architectural-msrs.html)[[69]](https://docs.kernel.org/admin-guide/hw-vuln/mds.html)：

所有 Intel CPU 缓冲区泄漏问题共享相同的缓解基础设施：
- 退出内核态时执行 `VERW` 指令
- 进入 VM 时执行 `VERW` 指令
- 空闲进入时执行 `VERW` 指令

```asm
; VERW 清除序列
; 使用特定的选择器以确保微码执行缓冲清除
mov     %ds, %rax            ; 保存 DS
mov     $__KERNEL_DS, %rdx   ; 加载内核数据段选择器
verw    (%rdx)               ; 执行 VERW 清除
mov     %rax, %ds            ; 恢复 DS
```

**版本演进**：
- MDS (2019)：初始 VERW 清除实现
- TAA (2019)：扩展 TSX 清除
- SRBDS (2020)：与 MDS 共享 VERW 缓解
- RFDS (2024)：新增对 Atom 特定的 VERW mds_buff 清除

**内核命令行**[[69]](https://docs.kernel.org/admin-guide/hw-vuln/mds.html)：
```
mds=full            # 完整 MDS 缓解（默认）
mds=full,nosmt      # 完整缓解 + 禁用 SMT
mds=off             # 禁用 MDS 缓解

tsx_async_abort=full           # 完整 TAA 缓解（默认）
tsx_async_abort=full,nosmt     # 完整 + SMT 禁用
tsx_async_abort=off            # 禁用 TAA 缓解

reg_file_data_sampling=on      # 启用 RFDS 缓解（默认）
reg_file_data_sampling=off     # 禁用 RFDS 缓解
```

**SMT 控制**[[68]](https://docs.kernel.org/admin-guide/hw-vuln/core-scheduling.html)：
- 对于 MFBDS、MLPDS 和 RFDS，SMT 是跨线程攻击向量
- 内核提供 `/sys/devices/system/cpu/smt/control` 接口控制 SMT
- 安全管理最佳实践：在受影响处理器上禁用 SMT

### 6.9 用户态 API：prctl() 推测控制

Linux 提供了 `prctl()` 系统调用接口，允许用户程序控制自身的推测执行行为[[65]](https://docs.kernel.org/userspace-api/spec_ctrl.html)：

```c
#include <sys/prctl.h>

// 获取当前投机控制状态
int state = prctl(PR_GET_SPECULATION_CTRL, PR_SPEC_STORE_BYPASS, 0, 0, 0);

// 设置投机控制
// PR_SPEC_ENABLE     - 启用保护（禁用投机）
// PR_SPEC_DISABLE    - 禁用保护（允许投机）
// PR_SPEC_FORCE_DISABLE - 强行禁用（不可逆）
prctl(PR_SET_SPECULATION_CTRL, PR_SPEC_STORE_BYPASS,
      PR_SPEC_ENABLE, 0, 0);

// Spectre v2 控制（间接分支推测）
prctl(PR_SET_SPECULATION_CTRL, PR_SPEC_INDIRECT_BRANCH,
      PR_SPEC_ENABLE, 0, 0);
```

**用户态推测控制影响**[[71]](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/include/uapi/linux/prctl.h)：
- 启用 `PR_SPEC_INDIRECT_BRANCH` 将导致：IBPB 刷新 + STIBP 启用
- 性能影响：进程运行速度降低 20-50%（因禁用间接分支预测）
- 推荐场景：密钥服务器、密码学守护进程、隔离沙箱

## 7. 编译器级防御机制

### 7.1 GCC 投机屏障支持

GCC 从 8.0 版本开始支持投机执行防御相关的编译器选项[[72]](https://gcc.gnu.org/gcc-8/changes.html)。

#### 7.1.1 Retpoline 支持

```bash
# GCC Retpoline 编译选项
-mindirect-branch=thunk          # 对间接调用使用返回梯形
-mindirect-branch=thunk-inline   # 代码内联返回梯形（非共享）
-mindirect-branch=thunk-extern   # 外部共享返回梯形（默认推荐）
-mindirect-branch-register       # 也对间接跳转使用返回梯形
-mfunction-return=thunk          # 函数返回使用返回梯形
-mfunction-return=thunk-inline   # 内联函数返回梯形
-mfunction-return=thunk-extern   # 外部函数返回梯形
```

**反向移植**：GCC 7.3/6.4/5.5 也通过发行版维护者反向移植了 retpoline 支持[[73]](https://gcc.gnu.org/legacy-ml/gcc/2018-01/index.html)。

#### 7.1.2 推测加载强化 (Speculative Load Hardening)

```bash
# GCC 8+ 的推测加载强化选项（x86）
-mspeculative-load-hardening    # 在条件分支后插入保护代码
                                # 使用数据依赖屏障防止推测加载
```

**工作原理**[[74]](https://llvm.org/devmtg/2018-10/slides/Carruth-SpeculativeLoadHardening.pdf)：
```c
// 原始代码：
if (idx < arr_len) {
    val = arr[idx];  // 推测执行可能绕过此检查
}

// SLH 保护后（简化示意）：
if (idx < arr_len) {
    idx_safe = idx & (arr_len - 1);  // 掩码操作（数据依赖）
    val = arr[idx_safe];
}
// 或使用 cmov 条件移动指令
```

#### 7.1.3 内联汇编推测屏障

GCC 支持以下内联汇编推测屏障[[75]](https://gcc.gnu.org/onlinedocs/gcc/x86-Built-in-Functions.html)：

| 屏障 | GCC 支持 | 说明 |
|------|---------|------|
| `lfence` | 原生 | AMD 上为序列化屏障 |
| `__builtin_ia32_lfence()` | GCC 内置 | x86 LFENCE 屏障 |
| `asm volatile("lfence" ::: "memory")` | 通用 | 内联汇编方式 |
| `csdb` | ARM GCC | ARMv8 推测屏障 |
| `SB` | ARM GCC | ARMv8.5 推测屏障 |
| `dsb sy` / `isb` | ARM GCC | ARMv8 隐式推测屏障 |

### 7.2 Clang/LLVM 投机屏障支持

Clang/LLVM 从 5.0 版本开始支持相关防御特性。

#### 7.2.1 Retpoline 支持

```bash
# Clang/LLVM Retpoline 编译选项
-mretpoline                     # 启用返回梯形
-mretpoline-external-thunk      # 使用外部共享返回梯形
-mno-retpoline                  # 禁用返回梯形
```

LLVM 11.0+ 支持 Spectre v2 的完整 retpoline 缓解[[76]](https://releases.llvm.org/11.0.0/docs/ReleaseNotes.html)。

#### 7.2.2 推测加载强化

```bash
# LLVM 12+（x86）
-mspeculative-load-hardening    # x86 推测加载强化

# Apple LLVM 特性（仅 arm64e）
-mbranch-protection=standard    # BTI + PAC
```

#### 7.2.3 LLVM LVI 缓解（仅限 SGX）

```bash
# LLVM 10.0+，专用于 Intel SGX 飞地保护
-mlvi-cfi                       # LVI 控制流完整性防护
-mlvi-hardening                 # 完整 LVI 加载值注入防护
```

**工作原理**[[77]](https://clang.llvm.org/docs/LVIMitigation.html)：在每条可能被利用的加载操作后插入 `lfence` 指令，防止 LVI 攻击中的值注入被后续推测执行利用。

### 7.3 编译器内置屏障函数

各编译器提供`__builtin` 推测屏障原语：

| 编译器 | 内置函数 | 效果 |
|--------|---------|------|
| GCC 12+ | `__builtin_speculation_safe_value(value, fallback)` | 生成数据依赖保护代码 |
| Clang | 同 GCC 兼容 | 同上 |
| MSVC 16.5+ | `_ReadWriteBarrier()` + `__lfence()` | 推测屏障 + 序列化 |
| ARM Compiler 6 | `__schedule_barrier()` | 推测调度屏障 |

## 8. 虚拟化平台防御机制

### 8.1 KVM (Kernel-based Virtual Machine)

KVM 是 Linux 原生虚拟化方案，集成了完整的瞬态执行防御[[78]](https://docs.kernel.org/virt/kvm/index.html)。

#### 8.1.1 VM 边界的缓冲清除

| 操作 | 机制 | 缓解的漏洞 |
|------|------|-----------|
| **VMEntry** | L1D Flush（MSR） | L1TF |
| **VMEntry** | VERW（MSR mds_ctrl） | MDS, TAA, RFDS, SRBDS |
| **VMEntry** | RSB 填充 | Spectre v2 (RSB 下溢), Retbleed |
| **VMExit** | RSB 填充 | Spectre v2 |
| **VM 切换** | IBPB | Spectre v2 跨 VM |
| **VMEntry** | BHI_DIS_S / BHB 清除 | BHI |

#### 8.1.2 VM 内的推测控制传递

KVM 向客户机传递主机推测控制能力[[79]](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/arch/x86/kvm/cpuid.c)：

```c
// KVM 的 CPUID 传递逻辑
// 主机上的 IA32_ARCH_CAPABILITIES 位被过滤后传递给客户机
// 客户机可以使用 prctl() 控制自身推测行为
kvm_cpuid_filter_arch_capabilities(...)
```

**可用能力传递**：
- `KVM_FEATURE_SPEC_CTRL`：客户机可访问 MSR 0x48/0x49
- IBRS/STIBP/IBPB 指令可用性传递
- SSBD 可用性传递
- PSFD (AMD) 可用性传递
- GDS_NO, RFDS_NO 等状态传递

#### 8.1.3 核心调度 (Core Scheduling) 集成

```bash
# 内核配置
CONFIG_SCHED_CORE=y

# 引导参数
sched_core=1
# 或通过 sysfs
echo 1 > /sys/kernel/debug/sched/core_sched
```

核心调度确保互不信任的 VM 或进程不会在 SMT 兄弟线程上同时运行[[68]](https://docs.kernel.org/admin-guide/hw-vuln/core-scheduling.html)。

## 9. Web 浏览器防御机制

### 9.1 共享内存与高精度定时器降级

#### 9.1.1 SharedArrayBuffer 限制

浏览器中主要的瞬态执行攻击向量通过高精度定时器和共享内存实现[[83]](https://security.googleblog.com/2018/07/mitigating-spectre-with-site-isolation.html)。

**Google Chrome**[[84]](https://www.chromium.org/developers/design-documents/site-isolation/)：
- 2021-2023 年间多次禁用 `SharedArrayBuffer`，直到引入跨域隔离策略
- 2023 年后要求启用"跨域隔离"（Cross-Origin Isolation）的页面才可访问 `SharedArrayBuffer`
- COOP: Cross-Origin Opener Policy (`Cross-Origin-Opener-Policy: same-origin`)
- COEP: Cross-Origin Embedder Policy (`Cross-Origin-Embedder-Policy: require-corp`)

**Mozilla Firefox**[[85]](https://wiki.mozilla.org/Project_Fission)：
- 65 版本 (2019)：禁用 `SharedArrayBuffer`，在 Spectre 缓解明确后重启用
- 79 版本 (2020)：重新启用 `SharedArrayBuffer`，同时要求 COOP/COEP 头

**Apple Safari**[[86]](https://webkit.org/)：
- Safari 15.2+ 启用跨域隔离支持 (COOP/COEP)
- 默认禁用 `SharedArrayBuffer` 直到启用跨域隔离

#### 9.1.2 定时器降级 (Timer Resolution)

| 浏览器 | 定时器精度 | 降级措施 |
|--------|-----------|----------|
| Chrome | 5μs 默认 → 100μs (降级) | 检测到潜在攻击时自动降级[[84]](https://www.chromium.org/developers/design-documents/site-isolation/) |
| Firefox | 5μs 默认 | 2018 年起默认使用 `performance.now()` 2ms 精度[[85]](https://wiki.mozilla.org/Project_Fission) |
| Safari | 1ms 精度 | 默认 1ms 精度，无进一步降级[[86]](https://webkit.org/) |
| Edge | 同 Chrome | 继承 Chromium 机制[[87]](https://learn.microsoft.com/en-us/deployedge/microsoft-edge-security-isolation) |

### 9.2 站点隔离 (Site Isolation)

**Google Chrome**[[84]](https://www.chromium.org/developers/design-documents/site-isolation/)：
- Chrome 67+ (2018)：默认启用严格站点隔离
- 每个渲染进程仅包含来自同一站点的页面
- 防止 Spectre 类型的跨站数据窃取
- 内存开销：+10-20% per renderer

**Mozilla Firefox (Project Fission)**[[85]](https://wiki.mozilla.org/Project_Fission)：
- Firefox 94+ (2021)：默认启用 Fission 站点隔离
- 每个站点独立渲染进程
- 防止跨站点瞬态执行攻击

**Apple Safari**[[86]](https://webkit.org/)：
- 使用 per-origin 进程模型
- Safari 15+ 增强隔离策略

### 9.3 JIT 编译器缓解

**Chrome V8 JIT**[[88]](https://v8.dev/blog/spectre)：
1. **索引掩码 (Index Masking)**：
   - 在数组边界检查处插入数据和推测双重掩码保护
   - 利用 `CMOV` 指令防止推测性越界访问

2. **推测污染追踪 (Speculative Poisoning, 实验性)**：
   - 将推测路径上获取的值标记为"受污染"
   - 阻止污染值作为后续内存访问的地址使用

3. **LFENCE 插入**：
   - 在某些关键推测屏障点插入 `lfence` 指令
   - 使用 Intel `lfence` 的序列化特性防止 Spectre v1

4. **Retpoline**：
   - 间址调用通过返回跳板重定向
   - V8 embedder 可选择编译选项

**Firefox SpiderMonkey JIT**[[89]](https://firefox-source-docs.mozilla.org/js/index.html)：
- 使用索引掩码保护数组访问
- 关键路径上的推测屏障
- 与 Firefox 沙箱机制配合

**Safari JavaScriptCore (JSC)**[[86]](https://webkit.org/)：
- 边界检查强化
- 推测性加载保护
- 与 iOS/macOS 内核 KPTI 配合

### 9.4 WebAssembly 防护

WebAssembly 通过以下机制防御瞬态执行攻击[[90]](https://webassembly.org/docs/security/)：

1. **线性内存模型**：Wasm 内存限制在沙箱内，无法访问主机内存
2. **无共享内存（默认）**：Wasm 默认不支持 `SharedArrayBuffer`
3. **确定性执行**：Wasm 语义限制可被利用的推测路径
4. **CSP (Content Security Policy)**：限制 Wasm 来源

## 10. 其他关键软件防御机制

### 10.1 数据库系统防御

#### 10.1.1 PostgreSQL

PostgreSQL 的瞬态执行防御[[91]](https://www.postgresql.org/docs/15/release-15.html)：
- 使用编译器 `-mspeculative-load-hardening` 选项编译关键路径
- 密钥管理模块 (SSL/认证) 使用 `prctl()` 禁用间接分支推测
- PAM/SSPI 认证模块强制 `PR_SPEC_ENABLE`

#### 10.1.2 MySQL / MariaDB

MySQL 的瞬态执行防御[[92]](https://dev.mysql.com/doc/refman/8.0/en/security.html)：
- TLS/SSL 密钥处理模块启用了推测控制
- 在 `mysqld` 初始化时通过环境变量传递 `PR_SET_SPECULATION_CTRL`
- 使用了 GCC/Clang retpoline 编译选项

#### 10.1.3 SQLite / 嵌入式数据库

SQLite 的瞬态执行防御[[93]](https://www.sqlite.org/security.html)：
- 无内置瞬态执行缓解策略
- 依赖操作系统内核提供的透明保护
- 建议在敏感部署中使用编译器推测加载强化编译

### 10.2 密码学库防御

#### 10.2.1 OpenSSL

**OpenSSL 3.x 的瞬态执行防御**[[94]](https://www.openssl.org/policies/secpolicy.html)：

1. **常量时间算法**：
   - RSA/AES/DH/ECDH/EdDSA 使用数据独立的执行路径
   - 禁止使用内存访问模式依赖于密钥的算法

2. **秘密数据清除**：
   - 使用 `OPENSSL_cleanse()` 显式清除敏感数据
   - 防止残留数据通过缓冲区泄漏攻击窃取

3. **推测屏障宏**：
   ```c
   // OpenSSL 内部推测屏障
   #define ossl_speculation_barrier() \
       __asm__ volatile("" ::: "memory")
   ```

4. **编译选项**：
   - 推荐使用 GCC/Clang retpoline 选项编译

#### 10.2.2 BoringSSL (Google)

BoringSSL 的瞬态执行防御[[95]](https://boringssl.googlesource.com/boringssl/)：
- 所有算法使用严格的常量时间实现
- 集成 Google 的推测屏障原语
- ARM 平台使用 `__asm__ __volatile__("dsb sy\nisb\n" ::: "memory")` 推测屏障
- x86 平台使用 `__asm__ __volatile__("lfence" ::: "memory")` 推测屏障

#### 10.2.3 libsodium

libsodium 的瞬态执行防御[[96]](https://doc.libsodium.org/)：
- 所有公开 API 使用常量时间实现
- 推测屏障使用编译器内存屏障 `__asm__ __volatile__("" ::: "memory")`
- 支持 WASM 环境的安全降级

### 10.3 容器运行时防御

#### 10.3.1 Docker / containerd

Docker 容器的瞬态执行防御[[97]](https://docs.docker.com/engine/security/)：
- 继承宿主机内核的所有瞬态执行防御
- 容器不具备独立的推测控制能力（共享内核）
- 可通过 `--security-opt seccomp=unconfined` 授予容器访问 `prctl()` 推测控制
- 推荐使用 seccomp 配置文件限制容器内可用系统调用

#### 10.3.2 Kubernetes

Kubernetes 中的瞬态执行防御[[98]](https://kubernetes.io/docs/tutorials/security/seccomp/)：
- 通过节点选择器 (NodeSelector) 将敏感工作负载调度到已修补的新架构硬件
- 使用 seccomp 配置文件限制容器系统调用
- `securityContext` 中的 seccomp 配置

```yaml
apiVersion: v1
kind: Pod
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: secure-container
    image: myapp:latest
```

#### 10.3.3 gVisor

gVisor (Google) 提供了更强的容器隔离[[99]](https://gvisor.dev/docs/architecture_guide/security/)：
- 用户态内核实现（Sentry），完全隔离宿主内核
- 即使宿主内核有瞬态执行漏洞，攻击者也难以渗透 Sentry
- 作为沙箱运行，大幅缩小攻击面

### 10.4 编程语言运行时防御

| 运行时 | 防御机制 | 版本 |
|--------|----------|------|

| **.NET (CoreCLR)** | JIT 推测屏障；retpoline 调用重写 | .NET Core 3.0+ [[101]](https://devblogs.microsoft.com/dotnet/) |

## 12. 结论与展望

### 12.1 当前防护态势总结

自 2018 年 Spectre/Meltdown 披露以来，整个计算生态系统在瞬态执行漏洞防御方面取得了显著进展：

**硬件层面**：
- Intel 通过 10+ 次微架构迭代逐步将安全特性硬化到硅片中，从 Coffee Lake 的 Meltdown 修复到 Granite Rapids 的 IPRED 和 DDP
- AMD 在 Zen 4 中实现了 Retbleed 的硬件修复，并在 Zen 5 中进一步增强了推测控制机制
- ARM 通过架构设计的先发优势（CSV2 系列原语）在大多数 ARMv8.2+ 处理器上提供了原生的硬件隔离

**软件层面**：
- Linux 内核建立了完善的推测控制框架 (KPTI + Retpoline/IBRS + VERW + IBPB + Core Scheduling)
- 编译器工具链 (GCC/Clang/MSVC) 全面支持推测屏障和返回跳板
- 浏览器通过站点隔离和定时器降级大幅缩小了 JavaScript 攻击面
- 密码学库普遍采用常量时间实现和推测屏障插入

### 12.2 依然存在的挑战

1. **SMT 困境**：同时多线程仍然是瞬态执行攻击的主要通道。完全禁用 SMT 的性能代价巨大（15-30%），Core Scheduling 缓解了但不能完全消除风险 [[68]](https://docs.kernel.org/admin-guide/hw-vuln/core-scheduling.html)

2. **性能退化累积效应**：逐层叠加的防御机制在旧款硬件上可导致 30-60% 的性能下降，迫使组织在安全和性能之间做出艰难权衡 [[54]](https://www.redhat.com/en/blog/performance-implications-spectre-and-meltdown-mitigations)[[60]](https://doi.org/10.1109/MICRO.2018.00032)

3. **IoT/嵌入式缺口**：大量 ARM 嵌入式设备运行老旧内核且无法获得微码更新，处于持续易受攻击的状态 [[107]](https://cwe.mitre.org/data/definitions/1427.html)

4. **新攻击面**：虚拟化层 (VMM)、可信执行环境 (TEE)、硬件加速器 (GPU/NPU) 的推测执行行为研究仍不充分 [[108]](https://mdsattacks.com/)

5. **形式化验证缺失**：目前缺乏端到端的形式化验证方法能够证明一个系统在所有推测状态下都是安全的 [[109]](https://arxiv.org/abs/2006.03841)

### 12.3 未来方向

1. **硬件层根本性修复**：未来微架构应从设计阶段将推测隔离纳入核心规格，如 CHERI 能力硬件、抗侧信道预取器、物理隔离的预测器表

2. **编译器自动化**：编译器应能自动识别安全关键的代码模式并插入最优的投机屏障（不以开发者手动标注为准）

3. **操作系统智能调度**：基于机器学习的工作负载分类，智能地选择最适合每一特定进程的防御强度 [[106]](https://www.phoronix.com/review/spectre-meltdown-2023)

4. **整机认证标准**：推动形成统一的瞬态执行安全认证标准（类似 Common Criteria），覆盖从硬件到应用的完整信任链

5. **开放硬件验证**：鼓励 RISC-V 等开放架构将硬件安全验证结果公开，实现真正可审计的安全性 [[110]](https://riscv.org/technical/specifications/)
