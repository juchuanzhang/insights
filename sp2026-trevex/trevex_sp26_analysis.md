# TREVEX 论文深度解析报告

> **论文标题**: TREVEX: A Black-Box Detection Framework For Data-Flow Transient Execution Vulnerabilities  
> **作者**: Daniel Weber, Fabian Thomas, Leon Trampert, Ruiyi Zhang, Michael Schwarz (CISPA Helmholtz Center for Information Security)  
> **发表于**: IEEE S&P 2026 (Artifact Evaluated, Available, Functional, Reproduced)  
> **代码**: https://github.com/cispa/trevex

---

## 1. 论文概述

### 1.1 研究背景

瞬态执行攻击（Transient Execution Attacks, TEA）自2018年 Meltdown 和 Spectre 被发现以来，持续对现代计算机系统构成严重安全威胁。多年来，尽管研究界投入了大量精力试图自动化瞬态执行漏洞（Transient Execution Vulnerabilities, TEV）的发现，绝大多数 TEV 仍然是通过手动方式发现的。现有的自动化方法存在严重局限：

- **基于变异已知攻击的方法**（如 Transynther/Medusa）：只能发现已知攻击的变体，本质上将现有假设投射到未来漏洞上，严重限制了发现范围。
- **基于泄漏契约/ISA 模拟器的方法**（如 Revizor）：需要精确建模已知泄漏行为，依赖模拟器的正确性和完整性。然而即使是最先进的 ISA 模拟器也往往不完整，无法正确模拟所有指令，更无法处理商业 CPU 中常见的未文档化指令。
- **基于 RTL 的方法**：需要访问 CPU 的寄存器传输级（RTL）描述，而商业 CPU 的 RTL 通常不公开，使得独立研究无法进行。

### 1.2 核心贡献

论文提出 **TREVEX**（TRansient Execution Vulnerability EXplorer），一个后硅（post-silicon）黑盒检测框架，用于自动发现数据流瞬态执行漏洞。核心贡献包括：

1. **框架设计与实现**：设计了不依赖 RTL、ISA 语义或泄漏契约的黑盒检测框架，可在 20 种 x86 微架构（Intel/AMD/Zhaoxin）上运行。
2. **发现 FP-DSS**：首次发现 **Floating Point Divider State Sampling**，影响 AMD Zen 和 Zen+ 微架构的新型 TEA，并演示了从 Linux 内核和 Chrome 浏览器泄漏数据。
3. **发现 Zero-at-ret 实例、新 FPVI 变体和 Zhaoxin FPVI**：在 Intel CPU 上发现 3 个 Zero-at-ret 实例（LVI-NULL 变体，受影响 CPU 列表此前未文档化）；在 AMD CPU 上发现不需要 denormal 输入值的新 FPVI 变体；首次发现 Zhaoxin CPU 受 FPVI 影响。
4. **全面回归测试**：重新发现了所有预期已知的数据流 TEV，包括此前工具无法发现的 GDS 和 FPVI。

### 1.3 负责披露

- **FP-DSS 和新 FPVI 变体**：向 AMD 披露，AMD 确认并分配 CVE-2025-54505，初始禁令期至 2026年3月16日，后延长至4月17日。
- **Zero-at-ret**：向 Intel 披露，Intel 正在内部重新评估该漏洞的安全影响。
- **Zhaoxin FPVI**：向 Zhaoxin 披露。

## 2. 分类体系（Taxonomy）

论文提出了一个覆盖所有已知 TEA/TEV 的分类体系，这是理解其工作范围的关键：

### 2.1 控制流瞬态执行漏洞（CF-TEV）

- **本质**：操纵瞬态**控制流**，如通过操纵分支预测机制使处理器沿错误推测路径执行。
- **典型实例**：Spectre-PHT（条件分支误预测）、Spectre-BTB（间接分支毒化）、Spectre-RSB（返回目标毒化）。
- **特征**：滥用**已充分理解的 CPU 特性**，属于设计层面的功能而非 bug。

### 2.2 数据流瞬态执行漏洞（DF-TEV）

- **本质**：利用瞬态**数据流**的异常变化，如错误转发结果，导致泄漏不可访问数据或向另一进程瞬态注入数据。
- **两个子类**：
  - **数据泄漏型（Data-leakage）**：如 Meltdown-US、MDS 系列（ZombieLoad、RIDL、Fallout）、GDS、DSS。
  - **值注入型（Value-injection）**：如 LVI、FPVI、SCSB。
- **特征**：多数源于**未文档化和非预期的 CPU 行为**（如延迟异常处理），本质上是 CPU 设计中的 bug，而非特性。

### 2.3 训练属性分类

- **Training-based**：需要先前的误训练才能可靠触发（如 Spectre-PHT/BTB、PSF、SSB）。
- **Training-free**：不需要任何误训练即可可靠触发（如 Meltdown-US、MDS、Spectre-RSB）。

### 2.4 TREVEX 的范围

**TREVEX 专注于 training-free DF-TEV**。原因：
1. DF-TEV 类似经典软件 bug，难以枚举潜在脆弱行为，更适合后硅测试。
2. DF-TEA 占近年来瞬态执行攻击的大多数。
3. 其设计本身不受训练属性限制，理论上也能发现 training-based DF-TEV，但生成有效训练序列的代码策略是挑战性的未来工作。

## 3. TREVEX 设计详解

TREVEX 的设计基于 **3 个关键技术**（T1、T2、T3），分为 **3 个阶段**（测试生成、执行、分类）。

### 3.1 总体架构（三阶段流水线）

```
阶段1: 测试生成 → 阶段2: 执行与泄漏检测 → 阶段3: 结果分类
        (T1 指令阴影)     (T2 数据流分歧测试, T3 跨特权域数据依赖检测)
```

**阶段1 — 测试生成**：
- 生成攻击者代码、受害者代码和环境参数。
- 应用指令阴影（T1）来污染任意微架构缓冲区。

**阶段2 — 执行与泄漏检测**：
- 在不同特权域中并行执行攻击者和受害者代码。
- 使用隐蔽通道恢复瞬态域中的数据。
- 应用数据流分歧测试（T2）检测非预期的瞬态数据流。
- 应用跨特权域数据依赖检测（T3）判断泄漏是否源于受害者。

**阶段3 — 分类**：
- 使用启发式规则过滤假阳性。
- 分类已知 TEV 类型。
- 聚类未知行为。

### 3.2 T1：指令阴影（Instruction Shadowing）—— 微架构污染

#### 3.2.1 问题

先前工作 Transynther 使用**手动编写的固定例程**来污染**已知**的微架构缓冲区（如内存子系统中的 line-fill buffer、staging buffer）。如果某个微架构元素未被这些例程覆盖，Transynther 就无法检测到来自该元素的泄漏。

#### 3.2.2 核心直觉

**当一个指令可以从特定微架构缓冲区泄漏数据时，同一指令也很可能能够影响该缓冲区的内容。** 例如：内存加载指令既可以从 line-fill buffer 泄漏数据，也可以用来污染 line-fill buffer。类似的行为在 GDS 中也得到验证——当另一个进程使用相同的向量指令污染微架构状态时，GDS 泄漏更加稳定、更容易检测。

#### 3.2.3 机制

- TREVEX 为受害者代码生成与攻击者代码**相似**的指令，称为"指令阴影"。
- 两种相似性模式：
  - **严格相似（Strict similarity）**：受害者使用与攻击者异常触发部分**完全相同**的指令。
  - **宽松相似（Lax similarity）**：受害者使用与攻击者指令**同类别、同 ISA 扩展集**的指令。
- 通过指令阴影，TREVEX 可以污染**已知和未知**的微架构缓冲区，使此前不可观测的 TEV 变为可检测（如 FP-DSS 的发现）。

#### 3.2.4 与 FP-DSS 发现的关系

手动验证表明，FP-DSS 仅在浮点除法单元最近被使用后才暴露瞬态泄漏。在空闲的 Ubuntu 系统上，浮点除法单元压力过低，无法观测到泄漏。指令阴影正是让 TREVEX 能够通过受害者的除法指令持续污染浮点除法单元的中间缓冲区，从而发现了 FP-DSS。

### 3.3 T2：数据流分歧测试（Data-Flow Divergence Testing）—— 泄漏检测

#### 3.3.1 核心原理

**当瞬态数据流（微架构状态）与架构数据流（架构状态）发生分歧时，存在非预期的瞬态行为，可能被攻击者利用。** 这一原理同时覆盖了数据泄漏型和值注入型漏洞——即使是纯数据流分歧（不直接泄漏数据），也可能导致可利用行为（如 LVI、SCSB、FPVI）。

#### 3.3.2 算法详解

算法基于隐蔽通道恢复的值列表（即 Lookup Table, LUT），其中第 i 个条目被标记表示值 i 在瞬态域中被恢复。分四种情况：

| 场景 | 条件 | 含义 | 是否泄漏 |
|------|------|------|----------|
| (a) 微架构与架构结果相同 | LUT 中仅 1 个值，且无异常 | 寄存器在架构和微架构层面结果一致 | ✗ |
| (b) 架构异常但微架构仍有访问 | LUT 中仅 1 个值 + 异常 | 异常应阻止架构可见结果，但瞬态域仍有数据流 | ✓（如 Meltdown-US） |
| (c) 架构与微架构结果不同 | LUT 中 ≥2 个不同编码值 | 寄存器在架构层面只有一个值，但瞬态域出现多个值 → 至少一个瞬态路径结果与架构结果不同 | ✓（如 MDS/microcode assist） |
| (d) 架构与 ≥2 个微架构结果不同 | LUT 中 ≥3 个编码值 | 多个瞬态路径产生不同结果 | ✓ |

**关键推论**：两个或更多不同的恢复值**必定**表示至少一个错误的瞬态数据流，无论攻击者进程是否捕获了异常——因为架构上，寄存器在任一时刻只能包含一个值。

#### 3.3.3 优势

- **相比 Shesha/µRL**：那些工具仅报告瞬态窗口的存在，不判断窗口中是否有实际可利用的数据流分歧。TREVEX 直接检测数据流分歧，大幅减少假阳性。
- **相比 Transynther**：Transynther 仅报告终端异常后的瞬态编码数据，TREVEX 可检测非终端异常场景（如 microcode assist 导致的 MDS 泄漏）。
- **相比 MRT 方法**：不需要 ISA 模拟器提供的"真值"，适用于任意代码片段、未文档化指令和浮点指令。

### 3.4 T3：跨特权域数据依赖检测（Cross-Privilege Domain Data-Dependency Detection）

#### 3.4.1 目的

判断观察到的泄漏是否依赖于来自受害者特权域的数据。这对于区分数据泄漏型 TEV（跨特权域数据流）和值注入型 TEV（无跨特权域依赖）至关重要。

#### 3.4.2 算法详解

```
CheckForDependency(Taints, NoiseThreshold, AccRatio):
    1. 对每个污点 t ∈ Taints:
       - 以 t 作为受害者数据执行测试
       - 收集泄漏值 → leakage_for_taint[t]
    2. 对每个泄漏地图执行 RemoveNoise（移除低于噪声阈值的条目）
    3. SymmetricDifference: 对每个污点 t:
       - 计算其泄漏值与所有其他污点泄漏值的对称差
       - 仅保留"唯一"的泄漏值（仅在 t 的地图中出现，不在其他地图中出现）
    4. 统计有多少污点可以通过唯一泄漏值被识别（dependent_leakage）
    5. 若 dependent_leakage > len(Taints) × AccRatio → 标记为存在数据依赖
```

**关键设计决策**：
- `SymmetricDifference` 采用保守策略，优化低假阳性率，但牺牲了对仅泄漏部分信息的依赖的识别能力。
- 可通过修改 `SymmetricDifference` 使其在值存在于 ≥N 个其他地图时才移除，来检测更细粒度的泄漏。
- 此算法能检测泄漏值与污点之间存在**确定性映射**的所有 TEV，包括不泄漏精确值的 TEV（如 MDS、GDS、DSS、FP-DSS）。

## 4. TREVEX 实现详解

### 4.1 测试生成阶段

#### 4.1.1 攻击者代码生成（3 部分）

**第1部分 — 架构状态准备**：
- 寄存器用有效内存指针、边界条件值（0、-1、各种浮点编码）、随机数据填充。
- 内存指针随机偏移以触发非对齐内存访问。

**第2部分 — 异常触发**：
- 生成预期会触发异常的汇编指令（异常是触发 TEA 最常见的方式：故障内存加载、microcode assist 等）。
- 指令语料库基于 uops.info 的完整 x86 指令列表（**14 000 条指令**）。
- **关键特性**：TREVEX 不假设生成的指令有任何语义，因此可以发射甚至未文档化的指令（如果提供了合适的编码列表）。
- 代码生成**引导向**异常触发，但允许探索不触发异常的潜在脆弱模式。

**第3部分 — 缓存编码**：
- 使用 Flush+Reload 将 CPU 寄存器单字节编码到 CPU 缓存。
- 每次迭代仅编码**选定寄存器的选定字节**，以保持瞬态窗口尽可能小。
- 寄存器和字节选择随机化。

#### 4.1.2 受害者代码生成

- 紧循环（tight loop），持续运行直到被 TREVEX 停止。
- 循环内操作：
  - 手动编写的内存子系统污染例程（内存加载、存储、刷新）。
  - 指令阴影：发射与攻击者异常触发部分相似（严格或宽松）的指令。

#### 4.1.3 环境参数生成

- **页表修改**：修改攻击者代码访问的内存页的页表条目（PTE）位，如清除 User 位、设置 Dirty 位。使用修改版的 PTEditor 内核模块。
- **配置寄存器修改**：修改 MXCSR 等配置寄存器，如清除 bit 11 以取消掩码浮点下溢异常，引入更多异常触发点。
- **缓存/TLB 状态修改**：决定将访问内存加入或移出 CPU 数据缓存或 TLB。

### 4.2 执行与泄漏检测阶段

- **环境设置**：使用 PTEditor 修改页表条目，恢复挥发性修改（如 Dirty 位、缓存状态）在每次重复执行前。
- **代码执行**：攻击者代码不循环，执行可配置次数以减少噪声；受害者代码持续紧循环。
- **泄漏检测**：基于 T2（数据流分歧测试），检查隐蔽通道恢复的值列表。
- **数据依赖识别**：基于 T3，重复执行测试用例并改变受害者污点值。

### 4.3 分类阶段

- **启发式分类 API**：提供可扩展的规则集接口，供人类专家定义分类规则。
- **级联式应用**：规则以级联方式应用，每条规则仅处理前一条未分类的结果。
- **基础规则集**：
  - 已知漏洞分类：Meltdown-US、MDS、GDS、FPVI、LVI-NULL 等。
  - 未知行为聚类：将不符合已知类别的测试用例按行为特征聚类。
- **LVI-NULL 分类规则示例**：检查是否存在异常（终端或非终端）、是否涉及内存加载、是否转发值 '0'。

## 6. 发现的漏洞详解

### 6.1 FP-DSS（Floating Point Divider State Sampling）—— 新型漏洞

**影响范围**：AMD Zen (Ryzen 5 2500U) 和 Zen+ (Ryzen 5 3550H)。

**触发条件**：
- 攻击者执行非故障的 SSE 除法指令（如 DIVSD）或 AVX 除法指令（如 VDIVSD）。
- 受害者在同核或兄弟硬件线程上也执行除法指令。
- 攻击者的除法瞬态转发依赖于受害者除法输入的数据。

**关键特征**：
- SSE 除法仅泄漏来自 SSE 除法的受害者数据；AVX 除法仅泄漏来自 AVX 除法的受害者数据。AVX→SSE 和 SSE→AVX 跨类型泄漏未观测到。
- 现有 DSS 缓解措施（清除整数除法单元状态）**不能**阻止 FP-DSS，因为浮点除法单元与整数除法单元在 AMD CPU 上是分离的。
- 推测泄漏源为 AMD 浮点除法单元的**中间缓冲区**。

**CVE**：CVE-2025-54505

**安全影响演示**：
1. **跨进程泄漏**：无特权攻击者可从另一进程泄漏数据。
2. **内核泄漏**：可从 Linux 内核泄漏数据。
3. **浏览器攻击**：JavaScript 引擎（Firefox 和 Chrome）会发射受 FP-DSS 影响的指令，恶意网站可利用此从本机进程泄漏数据。使用 WebAssembly 的 `f64.div` 和 `Uint8Array` 访问编码结果，配合 Evict+Reload（8 MB 内存驱逐）。
4. **击键时序攻击**：将泄漏的存在性作为侧信道，监测 GUI 应用中的击键时序。

**隐蔽通道带宽**：
- 跨进程隐蔽通道（不依赖共享内存），每 25 000 CPU 周期传输 3 bit（使用 9 种不同除法结果编码，8 种编码 3 bit 数据，1 种标记新传输开始）。
- 传输速率：158.732 kbit/s，错误率：0.38%（50 kB 数据传输测试）。

### 6.2 无需 Denormal 输入值的 FPVI 变体 —— 新变体

**影响范围**：多个 AMD CPU，包括最新 Zen 5 微架构（Ryzen AI 9 HX 370）。

**触发条件**：向量指令（如 SSE 的 `MULPD XMM1, XMM2`）瞬态转发错误值，**不需要** denormal 浮点输入值——这与已知 FPVI 条件（依赖 denormal 输入值）冲突。

**安全影响**：与 FPVI 类似的利用原语，攻击者可注入错误的向量值到受害者应用中。

**披露状态**：AMD 确认这是触发 FPVI 的替代方式，将发布安全简报。Intel CPU 上无法复现。

### 6.3 Zero-at-ret（LVI-NULL 变体）—— 新实例

**影响范围**：多个 Intel CPU（Ice Lake, Tiger Lake, Ice Lake-SP 等），这些 CPU 在 Intel 公开文档中标记为不受 LVI-NULL 影响。

**触发条件**：清除页表条目中的 Access 位即可瞬态转发值 '0'。例如 `ADOX RSI, [RDX]` 在 [RDX] 对应页的 Access 位被清除时，瞬态转发 '0' 到 RSI。

**与 LVI-NULL 的区别**：Intel 确认从攻击者角度看，Zero-at-ret 与原始 LVI-NULL 的唯一区别是瞬态窗口的长度。

**SGX 利用验证**：间接跳转 gadget 攻击在 Intel Core i3-1005G1 上成功率 > 99.9%。攻击使 CPU 从地址 0（而非正确函数指针表地址）瞬态获取跳转目标，在 SGX 威胁模型下攻击者可映射地址 0，从而瞬态重定向控制流。

**披露状态**：Intel 正在内部重新评估 Zero-at-ret 的安全影响。

### 6.4 Zhaoxin FPVI —— 新受影响厂商

**影响范围**：Zhaoxin KX-U6780A (LuJiaZui 微架构)。

**触发条件**：VEX 编码指令和非 VEX 编码指令均受影响。例如 `SUBPD XMM1, XMM2`：当 XMM2 包含 denormal 浮点值时，SUBPD 瞬态将 XMM2 的值转发到 XMM1（而非计算减法结果）；当 XMM1 包含 denormal 值时，瞬态使用 XMM1 的原始值而非减法结果。

**意义**：FPVI 此前仅知影响 Intel 和 AMD，Zhaoxin 是新确认的受影响厂商。

### 6.5 已知漏洞重发现

| 漏洞 | 重发现状态 |
|------|------------|
| Meltdown-US | ✓ 所有受影响 CPU |
| MDS (含 VRS) | ✓ 所有受影响 CPU |
| LVI(-NULL) | ✓ 所有受影响 CPU |
| GDS | ✓ 所有受影响 CPU |
| FPVI | ✓ 所有测试 CPU（Intel 文档化受影响；AMD 最新微架构仍受影响） |
| Meltdown-CPL-REG | ✓（需禁用 FSGSBASE 的特定配置） |
| Foreshadow | ✓（验证性测试） |
| DSS | ✓（手动验证代码可触发，但无受影响 CPU 进行完整验证） |
| SSB | 设计上可检测，但当前代码生成策略不够复杂以自动触发 SSB |

**首次自动化发现**：TREVEX 是首个能自动化检测 GDS 和 FPVI 的工具（此前工具不支持这两种漏洞）。

## 7. 与微软 Revizor 系列工作的对比

> 本章专门分析 TREVEX 与微软研究院的 Revizor 框架系列工作之间的对比，涵盖方法论、技术路线、适用范围和优劣。

### 7.1 Revizor 系列工作概览

微软研究院的 Revizor 框架是一个基于**模型的关系测试（Model-based Relational Testing, MRT）**方法，用于发现 CPU 违反泄漏契约（leakage contract）的行为。其核心系列论文包括：

| 论文 | 年份 | 会议 | 核心贡献 |
|------|------|------|----------|
| Revizor v1 [17] (Oleksenko et al.) | 2022 | ASPLOS | 首次提出基于泄漏契约的黑盒测试框架，使用 Unicorn 引擎生成契约轨迹 |
| Speculation at Fault [18] (Hofmann et al.) | 2023 | USENIX Security | 扩展到 CPU 异常建模，发现 DSS，但排除了浮点/SIMD/CFI 指令 |
| Hide and Seek with Spectres [19] (Oleksenko et al.) | 2023 | IEEE S&P | 优化随机测试效率，高效发现推测信息泄漏 |
| Enter, Exit, Page Fault, Leak [20] (Oleksenko et al.) | 2025 | arXiv preprint | 测试隔离边界（页面故障等），加入 DynamoRIO 后端替代 Unicorn |

### 7.2 方法论根本差异

| 维度 | TREVEX | Revizor |
|------|---------|----------|
| **哲学基础** | **漏洞发现工具**——目标是无假设地发现任何非预期瞬态数据流行为 | **契约验证工具**——目标是测试 CPU 是否遵守给定的泄漏契约模型 |
| **是否需要先验模型** | **不需要**——不依赖泄漏契约、ISA 模拟器或 RTL | **需要**——依赖泄漏契约定义"合法"行为，依赖模拟器生成契约轨迹 |
| **对 CPU 的假设** | 黑盒，零语义假设，甚至可测试未文档化指令 | 需要模拟器正确模拟指令语义；模拟器不支持的指令（含未文档化指令）无法测试 |
| **检测逻辑** | 数据流分歧：比较瞬态域与架构域的数据流，任何分歧即标记 | 关系测试：比较契约轨迹与硬件轨迹，相同输入下轨迹不同即标记违约 |
| **Oracle 构造** | 内置：数据流分歧即 oracle | 外置：泄漏契约即 oracle |

### 7.3 技术路线对比

#### 7.3.1 模拟器依赖 vs 无模拟器

| 方面 | TREVEX | Revizor |
|------|---------|----------|
| **ISA 模拟** | 不使用任何模拟器 | v1-v3 使用 Unicorn 引擎（基于 QEMU）；v2.0.0 加入 DynamoRIO 动态二进制插桩后端 |
| **浮点指令支持** | ✓ 完全支持（正是发现 FP-DSS 的基础） | ✗ Hofmann et al. [18] 明确排除浮点指令 |
| **SIMD 指令支持** | ✓ 完全支持 | ✗ 排除 SIMD 指令 |
| **CFI 指令支持** | ✓ 可测试 | ✗ 排除 CFI 指令 |
| **未文档化指令** | ✓ 可测试（提供编码列表即可） | ✗ 模拟器自然不支持 |
| **DynamoRIO 后端** | 不适用 | v2.0.0 加入，可绕过 Unicorn 的指令集限制，但仍受 DynamoRIO 插桩能力的限制 |

**关键结论**：TREVEX 在指令覆盖范围上有本质优势。Revizor 的 Unicorn 后端无法模拟大量 x86 指令（14000 条中仅小部分），使得许多潜在漏洞（如浮点除法单元相关的 FP-DSS）完全不在其测试范围内。即使 DynamoRIO 后端缓解了部分限制，它仍是一种动态插桩方法，无法覆盖所有微架构条件。

#### 7.3.2 微架构状态污染

| 方面 | TREVEX | Revizor |
|------|---------|----------|
| **污染机制** | 指令阴影：攻击者和受害者使用相似指令，自动污染任意（含未知）微架构缓冲区 | 无专门污染机制；依赖测试用例的随机输入配置来暴露泄漏行为 |
| **跨特权域污染** | ✓ 受害者在不同特权域执行，污染来自跨特权域 | 无专门的跨特权域执行设计 |
| **已知 vs 未知缓冲区** | 可污染已知和未知缓冲区 | 仅依赖契约模型定义的行为，未知缓冲区不在契约范围内 |

#### 7.3.3 泄漏检测机制

| 方面 | TREVEX | Revizor |
|------|---------|----------|
| **检测方法** | 数据流分歧测试：直接比较瞬态与架构数据流 | 关系测试：比较契约轨迹与硬件轨迹的观测可区分性 |
| **检测目标** | 任何瞬态数据流分歧（含值注入和数据泄漏） | 违反泄漏契约的观测差异（主要面向数据泄漏） |
| **值注入 TEV** | ✓ 可检测（FPVI、LVI 等） | ✗ 不直接面向值注入场景（契约模型通常不涵盖值注入） |
| **假阳性控制** | 数据流分歧 + 启发式分类 + 数据依赖检测 | 契约轨迹匹配控制；假阳性取决于契约的完备性 |
| **分类能力** | ✓ 启发式规则 + 聚类，自动分类已知/未知漏洞类型 | 仅报告契约违约，不分类漏洞类型 |

#### 7.3.4 测试用例生成

| 方面 | TREVEX | Revizor |
|------|---------|----------|
| **指令语料库** | uops.info 14000 条完整 x86 指令集 | 受模拟器/插桩能力限制的指令子集 |
| **生成策略** | 模板式 + 随机变异（类似语法引导 fuzzing），保持 TEA 总体结构 | 随机生成测试用例，基于契约约束输入域 |
| **环境参数** | 页表位修改、MXCSR 等配置寄存器修改、缓存/TLB 状态控制 | 页面故障等隔离边界测试（Enter-Exit-SP26） |
| **代码结构** | 3段式（状态准备 → 异常触发 → 缓存编码） | 多种执行域配置（用户态、内核态、SGX 等） |

### 7.4 发现能力对比

| 漏洞 | TREVEX | Revizor |
|------|---------|----------|
| Meltdown-US | ✓ 重发现 | ✓ 可发现（契约含已知泄漏模型时） |
| MDS | ✓ 重发现 | ✓ Hofmann et al. 发现了 DSS |
| DSS | ✓ 可触发 | ✓ 发现（Hofmann et al. [18]） |
| GDS | ✓ 首次自动化发现 | ✗ 模拟器不支持相关向量指令 |
| FPVI | ✓ 首次自动化发现 | ✗ 排除浮点指令 |
| LVI(-NULL) | ✓ 重发现 | ✗ 契约模型通常不涵盖值注入 |
| FP-DSS | ✓ **首次发现** | ✗ 浮点除法指令不在测试范围 |
| Zero-at-ret | ✓ 发现新实例 | 不明确（取决于契约是否覆盖 access-bit assist） |
| Zhaoxin FPVI | ✓ 发现 | ✗ 模拟器对 Zhaoxin ISA 支持未知 |

### 7.5 适用场景互补性

| 场景 | 更适合的工具 |
|------|-------------|
| **CPU 契约合规性验证** | Revizor——可形式化验证 CPU 是否遵守特定泄漏模型 |
| **未知漏洞广域搜索** | TREVEX——无需先验假设，可发现全新类型的漏洞 |
| **回归测试** | 两者均适用；TREVEX 的分类能力更适合回归测试场景 |
| **Intel/AMD 商业 CPU 独立测试** | TREVEX——不需要 RTL 或模拟器，真正的黑盒 |
| **开源 CPU 设计验证** | Revizor——可有完整 RTL 和准确模拟器 |
| **值注入 TEV 检测** | TREVEX——设计覆盖值注入和数据泄漏两类 |
| **浮点/SIMD 漏洞** | TREVEX——完整支持 |
| **SGX/TEE 场景** | 两者均支持（TREVEX 可扩展受害者域；Revizor v2.0.0 加入了页面故障测试） |

### 7.6 核心差异总结

1. **Revizor 的根本优势**在于其**形式化基础**——可以基于泄漏契约对 CPU 进行合规性测试，为软件提供安全保证。但其范围严格受限于契约的定义和模拟器的正确性。
2. **TREVEX 的根本优势**在于其**无假设广域搜索**——不依赖任何先验模型，可发现契约和模拟器之外的漏洞类型（如浮点指令相关的 FP-DSS 和 FPVI）。但其无法提供形式化安全保证。
3. **两者本质上解决不同问题**：Revizor 回答"CPU 是否遵守给定的安全模型？"，TREVEX 回答"CPU 是否存在未预期的瞬态数据流行为？"
4. **在发现能力上**，TREVEX 明确超越了 Revizor 的当前范围——特别是浮点指令、SIMD 指令、值注入 TEV 和未文档化指令领域。论文指出，即使 Revizor v2.0.0 加入了 DynamoRIO 后端，它仍受限于插桩能力而非真正的语义无关。
5. **在方法论上**，两者形成互补：Revizor 适合有模型可参考的验证场景，TREVEX 适合无模型可参考的探索场景。理想的 CPU 安全评估流程应结合两者——先用 TREVEX 广域搜索发现潜在漏洞，再用 Revizor 的契约框架形式化定义和验证修复后的合规性。

## 8. 关键技术有效性验证

### 8.1 指令阴影的有效性

FP-DSS 的发现直接证明了指令阴影的关键作用：
- 在空闲系统上，浮点除法单元压力过低，无泄漏可观测。
- 指令阴影让受害者持续执行除法指令，污染浮点除法单元的中间缓冲区，使攻击者的瞬态除法可转发受害者数据。
- 无指令阴影 → 无 FP-DSS 发现。

### 8.2 数据流分歧测试的有效性

- 可同时检测数据泄漏型（Meltdown-US, MDS, GDS, FP-DSS）和值注入型（LVI, FPVI）TEV。
- 不需要 ISA 语义知识，可测试浮点指令和未文档化指令。
- 相比仅检测瞬态窗口的方法（Shesha, µRL），有效过滤了大量假阳性。

### 8.3 跨特权域数据依赖检测的有效性

- 成功标记数据泄漏型 TEV（MDS, GDS, FP-DSS），使分析师可优先检查高严重性漏洞。
- 在 AMD Ryzen 5 2500U/3550H 上标记了跨特权域数据依赖——尽管这些 CPU 此前不知道受数据泄漏型 TEV 影响——直接引导了 FP-DSS 的发现。

## 9. 局限性与未来工作

### 9.1 为何不发现 Spectre 类漏洞？

TREVEX 发现的漏洞全部属于数据流瞬态执行漏洞（DF-TEV），没有包含任何 Spectre 类控制流瞬态执行漏洞（CF-TEV）。这并非偶然遗漏，而是**设计上的有意选择**，原因有三层：

1. **CF-TEV 滥用的是已文档化、已充分理解的 CPU 特性**。分支预测机制（PHT、BTB、RSB）的行为通常由 CPU 厂商文档化，其可被攻击者利用的方式（误预测控制流）是设计层面的"功能"而非 bug。研究者可以枚举预测器的行为模式来手动分析和发现 CF-TEV，因此自动化 fuzzing 发现的边际收益较低。相反，DF-TEV 源于**未文档化、非预期的 CPU 行为**（如延迟异常处理、缓冲区残留数据转发），本质上是 CPU 设计中的 bug，无法枚举潜在脆弱行为，更适合 fuzzing 自动发现。

2. **检测逻辑差异**。CF-TEV 需要生成分支误训练代码序列来可靠触发（即 training-based），如 Spectre-PHT 需要先让分支预测器学习到特定路径再触发误预测。TREVEX 的 PoC 当前不生成训练序列，仅面向 training-free 漏洞，因此天然无法触发 Spectre 类的 training-based CF-TEV。

3. **DF-TEV 占近年来发现的多数**。论文统计指出，近年的新 TEA（如 MDS、LVI、FPVI、GDS、FLOP、LAP、DROW 等）几乎全部是 DF-TEA，而 CF-TEA 大多数在 2018 年 Spectre 首次披露后的早期阶段被发现。这说明 DF-TEV 是当前更迫切的自动化发现需求。

论文在 Section 3 和 Section 8 明确说明：虽然 TREVEX 的设计本身不受流类型限制（理论上可扩展到 CF-TEV），但生成有效的分支误训练代码序列是一个独立且挑战性的工程问题，且 CF-TEV 的预测器行为已被厂商文档化、更容易手动发现，因此将 CF-TEV 的自动化发现留给未来工作。

### 9.2 跨安全域测试的实际覆盖局限

论文在**设计层面**（Section 4.1）声称 TREVEX 的受害者接口可实例化为多种安全域：

- 另一用户态进程（跨进程泄漏）
- 内核模块（内核→用户态泄漏）
- TEE enclave（如 Intel SGX 泄漏）

但**实际 PoC 实现与评估**（Section 6.1）仅使用了**用户态进程↔用户态进程**这一最基础的场景，运行在同核超线程（sibling hardware thread）上。**没有任何用户态↔内核态、虚机↔宿主机、或 SGX↔外部的实际 fuzzing 测试。**

论文选择仅做跨进程测试的理由是可移植性——20 种 CPU 中部分不支持 SGX 或未启用虚拟化，跨进程场景无需特定硬件功能即可广泛部署。然而，这带来了以下实质性问题：

1. **T3（跨特权域数据依赖检测）的命名与实际不符**：实际检测的是"跨进程数据依赖"，而非真正的跨特权域。跨进程泄漏能间接关联到跨特权域风险，是基于经验推断（如 MDS 的根因在 SMT 同核共享微架构缓冲区，缓冲区数据来源是内核还是用户进程不影响泄漏机制），而非直接验证。

2. **FP-DSS 的内核泄漏和浏览器泄漏均为手动 PoC**：这些演示证明漏洞的严重性，但不是 TREVEX 自动发现流程的一部分。自动化 fuzzing 中检测到的仅是跨进程级别的数据依赖。

3. **缺失的安全域场景**：
   - **内核→用户态**：需要实现内核模块受害者（如通过 PTEditor 在内核空间写入污点数据），TREVEX 已有页表修改基础设施但未构建内核受害者。
   - **SGX→外部**：需要 SGX SDK 开发 enclave 受害者代码，论文威胁模型提到此场景但未实现。
   - **虚机→宿主机 / 宿主机→虚机**：需要虚拟化环境配置，论文未提及此场景。
   - **跨核泄漏**：TREVEX 仅测试同核超线程，未测试跨核场景（如 CrossTalk [45] 类泄漏），尽管某些微架构缓冲区可能跨核共享。

这一局限使得 TREVEX 的"跨特权域"声称在实际验证层面存在缺口——设计上的通用性承诺与实现上的单一场景之间存在落差。要真正覆盖跨安全域的自动化漏洞发现，需要为每种特权域组合开发对应的受害者实例化实现。

### 9.3 其他当前局限

1. **仅支持 x86**：模块化设计使得移植到 ARM/RISC-V 可行，但尚未实现。
2. **仅面向 DF-TEV**：不检测 CF-TEV（如 Spectre-BTB），但设计上可扩展（需生成分支预测误训练代码）。
3. **仅面向 training-free DF-TEV**：PoC 不生成训练序列，理论上设计不受此限制，但训练序列代码生成是挑战性的工程问题。
4. **代码生成复杂度限制**：无法自动触发 SSB 等需要复杂指令序列的漏洞；指令阴影假设污染和泄漏指令相似，可能漏掉指令类型根本不同的泄漏通道。

### 9.4 未来改进方向

- 扩展到 ARM/RISC-V 架构。
- 增加训练序列生成策略以覆盖 training-based DF-TEV。
- 支持任意指令对（类似 Osiris），扩大搜索范围（代价是运行时间大幅增加）。
- 更复杂的指令序列生成技术。

## 11. 元评审（Meta-Review）摘要

S&P 2026 程序委员会的元评审确认了论文的以下贡献和关注点：

**接受理由**：
1. 创建了新工具以推动未来研究，可系统化发现漏洞且易于跨微架构移植。
2. 发现了新漏洞和变体，特别是 FP-DSS，并提供了跨特权边界和浏览器攻击的 convincing 演示。
3. 填补了领域空白：缺乏不依赖 RTL/ISA/已知攻击变异的通用自动化技术。

**值得关注的问题**：
1. 代码生成策略有限，不自主发现 training-based DF-TEV 或需要复杂模式的 DF-TEV（如 SSB）。
2. 评估限于 x86 CPU，移植到 ARM/RISC-V 的有效性有待验证。
